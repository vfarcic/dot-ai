## MCP OAuth Authentication & User Identity

Adds enterprise-ready authentication with individual user identity tracking. Previously, all users shared a single static token (`DOT_AI_AUTH_TOKEN`) with no way to identify who performed operations, control individual access, or revoke specific users.

dot-ai now implements OAuth endpoints per the [MCP Authorization spec](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization), enabling browser-based login through Dex as the default OIDC provider. MCP clients (Claude Code, ChatGPT, Windsurf, VS Code Copilot) complete the full OAuth flow — discovery, dynamic client registration, PKCE authorization, and token exchange — without requiring any specific gateway or ingress. Each authenticated user gets a `UserIdentity` (userId, email, groups) extracted from the OAuth token, providing the foundation for audit trails and future RBAC enforcement. User management endpoints (`POST/GET/DELETE /api/v1/users`) allow creating, listing, and removing Dex static users via the Dex gRPC API with immediate effect — no pod restarts or Secret editing needed.

Static token authentication (`DOT_AI_AUTH_TOKEN`) remains as a permanent alternative for CI/CD pipelines, REST API consumers, and MCP clients without OAuth support. Both modes coexist via dual-mode middleware — JWT validation first, static token fallback. Dex ships as a Helm subchart with auto-generated admin credentials on `helm install`; for production, admins configure Dex connectors for their IdP (Google, GitHub, LDAP, SAML). A branded Dex login page provides a consistent dot-ai experience.

See the [Authentication Overview](https://devopstoolkit.ai/docs/mcp/setup/authentication), [Connector Configuration](https://devopstoolkit.ai/docs/mcp/setup/connectors), and [MCP Client Setup](https://devopstoolkit.ai/docs/mcp/) for details.
